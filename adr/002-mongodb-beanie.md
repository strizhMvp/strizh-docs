# ADR-002: MongoDB + Beanie + Motor вместо PostgreSQL + SQLAlchemy

**Статус:** Принято  
**Дата:** 2026-05-04

## Контекст

Нужно выбрать базу данных для backend-сервисов. Основная нагрузка — хранение тест-кейсов с динамически определяемым набором полей (Admin настраивает Template), логи LLM-взаимодействий, промпты и настройки.

## Решение

MongoDB 7 как основная БД для всех backend-сервисов. Три отдельные базы: `strizh_auth`, `strizh_generation`, `strizh_admin`. ODM — Beanie поверх Motor.

## Обоснование

Ключевой аргумент — `test_cases.fields`. Набор полей тест-кейса определяется Template-ом, который Admin меняет в рантайме. В PostgreSQL это либо JSONB-колонка (теряем типобезопасность), либо EAV-паттерн (кошмар для запросов). MongoDB хранит это естественно.

То же касается `prompts`, `settings`, `logs.payload` — всё документоориентированное.

**Beanie** даёт Pydantic-модели как схему документов (те же модели используются в FastAPI-роутерах), встроенный ODM без написания сырых Motor-запросов вручную.

**Motor** — официальный async-драйвер, поддерживается MongoDB Inc.

## Последствия

- Postgres всё равно используется — но только для Atlassian DC (Jira и Confluence требуют реляционную БД)
- Схема гибкая: добавление нового поля в Template не требует миграций
- Beanie не поддерживает JOIN — межколлекционные запросы решаем на уровне сервиса, не БД
