# ADR-001: Микросервисная архитектура с первого дня

**Статус:** Принято  
**Дата:** 2026-05-04

## Контекст

Проект разрабатывается двумя людьми с разными зонами ответственности: backend (Руслан) и frontend (Андрей). Нужно определить, начинать ли с монолита и позже разбивать, или сразу строить сервисы.

## Решение

4 backend-сервиса с первого дня, каждый в отдельном репозитории:

| Сервис | Порт | Ответственность |
|--------|------|-----------------|
| strizh-auth-service | 8001 | JWT, пользователи, PAT |
| strizh-generation-service | 8002 | LLM-оркестрация, SSE, история |
| strizh-integration-service | 8003 | Confluence, Jira, Xray |
| strizh-admin-service | 8004 | Шаблоны, промпты, настройки, логи |

## Обоснование

Границы между доменами очевидны с самого начала и не пересекаются. Разбивать монолит позже — дороже, чем начать правильно. Команда маленькая, но сервисы изолированы так, что не мешают друг другу при параллельной разработке.

## Последствия

- Межсервисная коммуникация — только REST через `/internal/*` эндпоинты
- nginx выступает API Gateway с `auth_request` для централизованной авторизации
- Каждый сервис деплоится и масштабируется независимо
